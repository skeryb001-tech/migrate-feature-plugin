# 前端功能迁移

用两个锚点约束整个迁移：

- **行为等价（parity）**：相同输入产生相同输出、状态转换、副作用、错误反馈、交互和视觉结果。
- **目标优先（target-first）**：内部实现复用目标能力，并直接遵循目标目录、模块分层、命名、请求、状态、样式和测试规范。

发生冲突时按以下顺序裁决：

1. 保持行为等价。
2. 复用目标已有实现。
3. 按目标规范和代码结构落盘。
4. 仅在兼容前三项时参考源目录和文件名。

## 运行方式

### 1. 建立迁移账本

保持源项目/源页面只读，先读取源与目标生效规范和工作区状态。选择一个模式：

- `cross-project`：不同仓库、独立工程、独立 package，或同仓库但构建与依赖边界独立。
- `cross-page`：同一工程内迁移到另一个页面、路由或业务入口。

从本 Skill 目录运行：

```bash
python3 scripts/createMigrationReport.py \
  --mode cross-project \
  --source <source> \
  --target <target> \
  --output <report.md>
```

报告必须在实现前创建。默认把它放在临时工作区；只有用户或目标规范要求时才写入产品仓库。

完成条件：报告包含正确模式、源/目标入口和完整 `PH1–PH9`、模式专项 Checklist。

### 2. 逐阶段执行

严格按报告中的九阶段顺序执行：

```text
梳理盘点 → 环境与目标映射 → 底层能力 → 业务逻辑
→ 页面与 UI → 上下文集成 → 多层校验 → 回归验收 → 归档
```

每个阶段先完成 legwork，再对照该阶段“完成条件”和门禁。当前阶段未满足时，留在当前阶段修复；后续验证暴露问题时，退回最早受影响阶段并重跑其后门禁。

执行阶段 8 前读取 [验收与证据规则](references/migration-contract.md)。遇到 P0/P1、校验失败或连续返工时，只读取 [高频故障兜底手册](references/failure-playbook.md) 的对应症状。

完成条件：报告中当前阶段和当前模式的每个稳定 ID 均已勾选，并附有可复现证据。

### 3. 运行硬门禁

阶段 1/2 和阶段 6 使用冲突扫描：

```bash
python3 scripts/scanMigrationConflicts.py <source-feature-root> <target-root>
```

阶段 8/9 使用报告校验：

```bash
python3 scripts/validateMigrationSpec.py
python3 scripts/validateMigrationReport.py <report.md>
```

退出码 `0` 才是完整 PASS：G0–G9 全部 PASS、P0=0、P1=0、归一化总分 ≥95。退出码 `3` 表示代码级完成但运行时视觉待验收，继续保留 G6/G8=`PENDING_RUNTIME`；退出码 `1` 表示报告或验收失败。

## 执行规则

### 资产与目标映射

- 追踪“入口 → handler → composable/hook → store → service/API → 状态更新/导航/埋点”，直到每个可观察结果都有源证据。
- 按“复用 → 适配 → 迁入 → 阻塞”处理能力。契约等价时复用；部分等价时建立最窄适配层；目标缺失时只迁入最小实现。
- 在阶段 2 确定每项源职责的目标落点，阶段 3 起直接按目标结构实现。阶段 6 只做上下文集成。
- 保护目标同名文件：职责等价时复用或最小合并；职责不等价时保留目标文件，并按目标规范语义化重命名迁入文件。
- 同步更新组件名、静态/动态导入、自动导入、路由、barrel export、types、tests 和 assets 引用。

### 逻辑、UI 与消费者

- 保持输入、输出、默认值、校验、状态机、请求参数/次数、错误、重试/取消、缓存、权限、支付、实验和埋点语义等价。
- 为请求、订阅、watcher、轮询、计时器、写入和埋点指定唯一职责所有者，并在离开作用域时完成清理。
- UI 结果以源为基准，实现方式使用目标组件、设计 token、i18n、图标和样式系统。
- 在真实浏览器中固定相同数据、视口、缩放、主题、语言和字体状态，记录引擎与版本，保留源/目标截图、CSSOM computed style 与几何测量；关键宽高、间距、对齐和位置误差必须 `≤1 CSS px`。
- 修改共享组件、store、composable、service、类型或导出前，扫描全部直接、间接、动态和自动导入消费者，并保持兼容或取得破坏性变更授权。

### 风险与证据

- **P0 阻断**：数据、安全、权限、生产配置、同名覆盖、错误写入或破坏目标公共能力。立即停止写入，解决并复验后继续。
- **P1 高危**：主流程、关键分支、关键 UI、业务语义或副作用次数错误。当前阶段保持未通过，修复后继续。
- **P2 常规**：不阻断核心使用的细节偏差。记录证据、扣分，并由用户接受后进入迭代清单。

保持目标已有改动完整。依赖新增、升级、降级、替换和公共契约破坏均以用户授权为准。环境或浏览器视觉证据缺失属于 `UNVERIFIED`，不属于 `N/A`；自定义渲染器不能替代浏览器门禁，此时结论必须为 `CODE_ONLY`，不能声明一比一验收通过。

有效证据必须能让另一位执行者复现，包括命令输出、测试、Network、调用链、截图、computed style、消费者搜索或 diff。文字判断本身不构成证据。

## 最终交付

交付迁移报告和代码改动，并总结：

- 模式、范围、源/目标基线和权威验收来源。
- 调用链、能力映射、目标结构、重命名和消费者影响。
- P0/P1/P2、验证命令、证据、归一化评分和结论。
- 用户可感知差异、已接受 P2、跳过项和残余风险。
- 回滚起点、入口关闭、公共模块/API/埋点恢复和持久化清理。

最终完成条件：报告校验返回 0，目标项目相关检查通过，源行为与目标既有消费者均无回归。
