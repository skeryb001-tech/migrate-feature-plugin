# 前端功能迁移

严格按“前置约束 → 标准化流程 → 分层执行 → 卡点校验 → 验收标准 → 故障兜底 → 归档规范”执行。

开始前必须读取 [迁移契约与验收规则](references/migration-contract.md)，选择迁移模式并只读取对应专项清单：

- 不同仓库、工程或 package 之间迁移：读取 [跨项目迁移清单](references/cross-project-checklist.md)。
- 同一项目的页面、路由或业务入口之间迁移：读取 [跨页面迁移清单](references/cross-page-checklist.md)。
- 出现风险、验证失败或连续返工：读取 [高频故障兜底手册](references/failure-playbook.md)。

## 1. 前置约束

### 1.1 固定决策优先级

遇到冲突时严格按以下顺序裁决，低优先级不得破坏高优先级：

1. **源行为一致**：迁移后的逻辑、数据契约、功能、交互、UI 状态和视觉结果与源功能一致。
2. **目标能力复用**：优先使用目标已有组件、composable/hook、store、service/API、utils、types、设计 token、i18n、资源和依赖。
3. **目标规范与结构**：迁移后的目录、分层、文件命名、编码风格、状态边界、请求方式和测试方式以目标项目为准。
4. **源结构与命名参考**：仅当不违反目标规范、没有同名/大小写/自动导入冲突且不会制造第二套能力时，才保留源相对结构和文件名。

把“一比一”定义为用户可观察行为一致，不是逐文件复制。相同输入必须得到相同输出、状态转换、副作用、错误表现、交互反馈和视觉结果；内部实现必须融入目标架构。

### 1.2 基础信息

- **适用场景**：只用于包含前端逻辑或 UI 的完整功能迁移；纯文案修改、单点样式调整、无 UI 后端数据迁移或普通重构不使用本 Skill。
- **执行角色**：前端或全栈主执行；API、权限、发布相关角色按边界参与；测试角色负责主流程、分支、边界和视觉验收。
- **默认写入边界**：源项目/源页面只读，目标项目/目标页面可写；除非用户明确授权，不修改源。
- **交付产物**：迁移报告、风险清单、调用链、能力/文件映射、代码改动、消费者清单、验证证据、评分、回滚方案和归档结论。
- **合格底线**：总分至少 95；P0/P1 未解决项为 0；全部硬门禁通过。

### 1.3 模式选择

开始阶段 1 前只选择一个主模式，并写入迁移报告：

| 模式 | 判定 | 额外保护对象 |
| --- | --- | --- |
| `cross-project` | 源与目标属于不同仓库、工程或独立 package | 目标架构、依赖边界、配置安全和目标既有消费者 |
| `cross-page` | 源与目标位于同一项目，只改变页面、路由或入口 | 源页面原行为、页面状态隔离、导航生命周期和共享消费者 |

跨 monorepo package 时按 `cross-project` 执行；同仓库但技术边界、构建入口和依赖清单独立时也按 `cross-project` 执行。

### 1.4 准入条件

进入实现前逐项确认：

- [ ] 源/目标路径、入口、基准分支或提交和生效的项目规范可读取。
- [ ] 已追踪真实调用链，并拥有逻辑与 UI 的源基准证据。
- [ ] 目标路由、布局、修改范围、包管理器、依赖和验证方式明确。
- [ ] API、鉴权、数据、权限、埋点、上传下载、缓存和持久化边界明确。
- [ ] 目标已有能力和同类实现完成仓库级搜索。
- [ ] 目标工作区已有改动已识别并受保护。
- [ ] 公共组件/store/composable/service 的全部消费者可扫描。
- [ ] 起始提交、入口关闭方式和公共改动回滚方式已记录。

优先从代码和环境发现信息。只有缺失信息会改变业务规则、数据契约、页面布局、依赖方案或验收基准时才提问，并暂停受影响阶段。

### 1.5 禁止执行条件

命中任一项立即停止写入：

- 无法读取源功能、目标位置或生效规范，不能建立真实调用链。
- 关键数据、权限、支付、生产配置或 API 契约存在未解决冲突。
- 准备覆盖目标同名文件、目录、路由、store、导出、组件名或自动导入名。
- 需要新增、升级、降级或替换依赖但未获授权。
- 目标文件包含无法安全合并的用户改动。
- 修改共享能力前无法确定消费者范围，或无法回归关键消费者。
- 没有可执行回滚方案。
- 缺少视觉基准却准备宣称 UI 一比一通过。
- 存在未解决 P0 或当前阶段 P1。

### 1.6 风险分级

| 等级 | 典型风险 | 强制动作 |
| --- | --- | --- |
| P0 阻断 | 同名覆盖、数据/权限冲突、生产配置泄漏、错误写入、破坏目标公共能力 | 立即停止；解决并复验前不得继续 |
| P1 高危 | 主流程或关键分支失败、逻辑/状态丢失、重复副作用、关键 UI 不可用 | 停止当前层；修复并通过门禁 |
| P2 常规 | 不阻断核心使用的样式、文案、动效或低频边界瑕疵 | 记录证据、扣分；仅用户接受后迭代 |

无法判断 P0/P1 时按更高等级处理。每项风险记录编号、证据、影响、动作、状态和复验结果。

## 2. 标准化流程

以下九阶段不可逆序。失败时退回最早受影响阶段，并重跑其后门禁：

| 阶段 | 固定动作 | 强制产出 |
| --- | --- | --- |
| 1. 梳理盘点 | 逆向源调用链；盘点 UI 状态、目标能力、文件和消费者 | 基线、调用链、UI 矩阵、复用候选、风险 |
| 2. 环境准备 | 对齐规范、包管理器、依赖、别名、配置和验证条件 | 差异表、目录/分层映射、验证与回滚方案 |
| 3. 底层能力落地 | 先完成 types、API、状态、基础 UI、token、资源与适配层 | 能力映射、底层验证 |
| 4. 业务逻辑落地 | 对齐校验、状态机、异步时序、权限、缓存、埋点和异常 | 逻辑契约与主/分支证据 |
| 5. 页面/功能复刻 | 复刻结构、文案、交互、全部 UI 状态和响应式 | 页面实现与 UI 对照 |
| 6. 差异化适配 | 按目标规范重组代码；处理路由、i18n、命名和冲突 | 最终路径映射、适配与重命名记录 |
| 7. 多层校验 | 执行静态、功能、视觉、边界、副作用和消费者回归 | 验证矩阵与修复证据 |
| 8. 回归验收 | 对照源基准和目标既有能力，计算客观评分 | 回归报告、评分与结论 |
| 9. 归档沉淀 | 汇总决策、文件、风险、证据、回滚和后续项 | 可追溯交付记录 |

每阶段必须在迁移报告中输出可勾选、可检查的 Checklist；当前阶段未通过，不得开始下一阶段。

## 3. 分层执行

### 3.1 资产与环境层

从用户入口追踪“页面/组件 → handler → composable/hook → store → service/API → 服务端/第三方 → 状态更新/导航/埋点”。不得停在薄页面或表层组件。

盘点路由、layout、middleware、page meta、UI 系统、token、i18n、权限、实验、SEO、缓存、持久化、上传下载、轮询、测试和配置。记录源提交与目标起始提交，避免基线漂移。

### 3.2 目标能力与结构层

按“复用 → 适配 → 迁入 → 阻塞”决策每项能力：

1. 按目标项目规定的优先级搜索 demo、共享 UI、业务组件、hook、store、service、utils、types 和依赖。
2. 契约等价则直接复用；部分不同则增加最窄适配层。
3. 目标缺失时只迁入最小必要实现，并放入目标规范指定的目录和分层。
4. 源目录仅用于理解职责和建立路径映射，不作为目标落盘依据。
5. 缺少数据、权限、资源或授权时标记阻塞，不用 mock、硬编码、吞错或静态数据伪装完成。

若目标同名文件存在：

1. 职责和契约等价：复用目标文件或最小合并，不复制源文件。
2. 职责不同：保留目标文件，按目标命名规范为迁入能力生成业务语义名称。
3. 禁止使用 `Copy`、`New`、`Temp`、`2` 等临时后缀。
4. 同步更新组件名、静态/动态导入、自动导入、路由、barrel export、类型、测试和资源引用。
5. 执行精确与大小写冲突扫描；有冲突时禁止进入实现。

### 3.3 业务逻辑层

保持输入、输出、默认值、校验、状态转换、异步顺序、请求参数/次数、错误、重试/取消、缓存、刷新恢复、权限、额度、支付、实验和埋点一致。

检查目标生命周期是否已经产生同类请求、订阅、计时器或埋点。迁移后必须只有明确的副作用所有者，并在页面离开或作用域销毁时完成清理。

### 3.4 页面与 UI 层

覆盖初始、加载、空、成功、失败、禁用、hover、active、focus、选中、弹层、长文案、极端数据、移动端、主题和 reduced-motion。

UI 视觉结果以源功能为基准，代码实现以目标 UI 组件、设计 token、i18n、图标和样式规范为准。固定相同数据、视口、缩放、语言、主题和字体状态，对比结构、盒模型、颜色/透明度、字体层级、边框、圆角、阴影、资源、动效、键盘和触控行为。

### 3.5 共享消费者与回滚层

修改目标公共组件、store、composable、service、类型或导出前：

1. 仓库级扫描直接、间接、动态和自动导入消费者。
2. 记录消费者清单、原契约、兼容策略和聚焦回归方式。
3. 默认保持向后兼容；必须破坏契约时升级为 P0 并等待明确授权。
4. 验证源入口、目标入口和所有关键既有消费者。

实现前写明回滚起点、入口/路由关闭方法、公共模块恢复方式、API/埋点恢复方式，以及缓存/持久化清理步骤。回滚不可执行时不得开始写入。

## 4. 卡点校验

| 门禁 | 通过条件 |
| --- | --- |
| G0 准入 | 模式、路径、范围、基准、规范、权限和回滚明确 |
| G1 资产 | 调用链、UI 状态、API/状态/权限/埋点和目标复用候选完整 |
| G2 冲突 | P0=0；文件、大小写、路由、store、导出和自动导入冲突为 0 |
| G3 结构 | 目录、模块分层、命名和实现方式符合目标项目规范 |
| G4 底层 | 类型、API、状态、资源和基础组件可独立工作 |
| G5 逻辑 | 主流程与关键分支一致，无重复副作用 |
| G6 UI | 关键状态完整，同条件对照无 P1 视觉差异 |
| G7 消费者 | 共享能力全部关键消费者回归通过 |
| G8 验收 | 静态、功能、视觉、边界验证完成且评分合格 |
| G9 归档 | 映射、风险、证据、评分、回滚和结论可追溯 |

每个门禁只允许“通过 / 不通过 / 受限通过”。P0/P1 不允许受限通过；P2 只有记录证据、扣分并获得用户接受后才允许受限通过。

## 5. 脚本化门禁

脚本均只使用 Python 标准库。以下路径相对于本 `SKILL.md` 所在目录；从 Skill 目录执行，或替换为脚本绝对路径，不要把脚本复制进目标项目。先查看 `--help`，再按实际路径执行：

```bash
python3 scripts/scanMigrationConflicts.py <source-feature-root> <target-root>
python3 scripts/createMigrationReport.py --mode cross-project --source <source> --target <target> --output <report.md>
python3 scripts/validateMigrationReport.py <report.md>
```

- 阶段 1/2 使用冲突扫描；返回非 0 表示存在不同内容同名、大小写或路径类型冲突。
- 阶段 1 创建模式专属迁移报告；默认不覆盖已有报告，只有明确需要时使用 `--force`。
- 阶段 8/9 校验报告；存在占位符、缺失证据、未通过硬门禁、P0/P1、错误计分或总分低于 95 时返回非 0。
- 脚本是最低门禁，不替代仓库级语义搜索、真实功能验证和视觉对照。

## 6. 验收标准

固定权重：静态代码 15、功能逻辑 35、视觉还原 30、边界场景 15、归档完整 5。

只有证据完整的通过项才得满分；未验证、失败和无证据项记 0；部分得分只允许用于已记录且用户接受的 P2。详细计分和默认 UI 误差按 [迁移契约与验收规则](references/migration-contract.md) 执行。

| 结论 | 条件 | 动作 |
| --- | --- | --- |
| 合格 | 95–100；P0=0、P1=0；全部硬门禁通过 | 可交付；已接受 P2 进入迭代清单 |
| 不合格 | 80–94，且无 P0/P1 | 修复后重跑阶段 7/8 |
| 重做 | <80，或任一 P0/P1/硬门禁失败 | 退回最早错误阶段并重做后续阶段 |

无法运行源或目标、缺少关键视觉基准或未做同条件对照时，只能标记“代码级迁移完成，运行时待验收”，不得判定一比一通过或给出 95 分以上结论。

## 7. 故障兜底与归档

出现问题时按“保护目标既有能力 → 冻结当前层 → 定位最早错误阶段 → 最小修复 → 重跑后续门禁”执行。不得通过覆盖、扩大重构、吞错、禁用校验、删除测试、临时后缀或修改无关代码绕过问题。

最终交付必须包含：

- 模式、范围、源/目标路径、基准提交和验收环境。
- 调用链、目标复用能力、最终目录/分层、文件重命名和引用映射。
- 共享消费者、兼容策略、P0/P1/P2 风险和复验结果。
- 静态、功能、视觉、边界验证的命令、环境、结果与证据。
- 客观评分、用户可感知差异、已接受 P2、跳过项和残余风险。
- 回滚起点、入口关闭、公共模块/API/埋点恢复和缓存清理步骤。

除非用户或目标规范要求，不把临时报告写入产品仓库。永久归档时遵循目标项目文档目录和命名规范，且不得包含密钥、cookie、token、个人数据或生产敏感响应。

不得主动暂存或提交。仅在用户明确要求时按目标仓库提交规范处理，并在提交前重跑 G8/G9。
