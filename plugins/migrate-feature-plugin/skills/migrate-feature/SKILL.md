# 前端跨项目与跨页面功能迁移

严格按“前置约束 → 标准化流程 → 分层执行 → 卡点校验 → 验收标准 → 故障兜底 → 归档规范”执行。开始任务时读取并填写 [迁移 SOP 与验收模板](references/migration-contract.md)；出现 P0/P1 风险、验证失败或反复返工时，再读取 [高频故障兜底手册](references/failure-playbook.md)。

把“一比一”定义为用户可观察行为一致，而不是逐文件复制：相同输入必须产生相同输出、状态转换、副作用、错误表现、交互反馈和视觉结果。允许内部实现遵循目标项目架构，但不得静默改变业务语义。

## 1. 前置约束

### 1.1 Skill 基础信息

- **定义**：把一个可独立识别的前端页面、模块、组件或完整业务流，从源页面/项目迁入目标页面/项目并完成适配与回归。
- **适用场景**：仅用于包含业务逻辑和 UI 的完整功能迁移；纯文案替换、单点样式微调、后端数据迁移或无 UI 的普通重构不使用本 Skill。
- **执行角色**：前端或全栈负责主迁移；涉及 API、权限或部署时纳入后端/运维边界；测试角色必须参与契约、分支、边界和视觉验收。
- **执行范围**：默认只读取源项目，只写入目标项目；除非用户明确要求，不修改源项目。
- **交付产物**：迁移契约、风险清单、源到目标能力映射、文件/目录映射、代码改动、验证证据、量化评分、差异清单和归档记录。
- **合格底线**：总分至少 95 分，P0/P1 未解决项为 0，主流程、关键分支、关键 UI 状态和命名冲突检查 100% 通过。

### 1.2 准入条件

进入实现前必须满足：

- [ ] 可读取源项目/页面、目标项目/页面和各自生效的 `AGENTS.md`、rules 与工程规范。
- [ ] 已定位源功能入口、真实调用链、基准分支/提交和至少一种验收证据。
- [ ] 已明确目标入口、路由、布局、允许修改范围和目标项目复用优先级。
- [ ] 已读取两个项目的包管理器、依赖、构建、UI、状态、请求、i18n 和测试约定。
- [ ] 已检查两个工作区状态，并区分用户已有改动与迁移改动。
- [ ] 已明确 API、权限、数据、埋点、上传下载、缓存与持久化边界。
- [ ] 已确认开发服务、浏览器验证、依赖变更和公共能力修改权限。

优先从代码和环境发现信息。只有缺失信息会改变数据契约、业务规则、页面布局、依赖方案或验收基准时才提问，并暂停受影响阶段。

### 1.3 禁止执行条件

命中任一项立即停止写代码：

- 源功能或目标位置不可读，无法建立真实调用链。
- 关键数据、权限、支付、生产配置或 API 契约存在未解决冲突。
- 准备覆盖目标项目已有同名文件、目录、路由、store、导出或自动导入名。
- 需要新增/升级/替换依赖，但未获得用户授权。
- 目标文件包含无法安全合并的用户改动。
- 缺少视觉基准却要求宣称 UI 一比一验收通过。
- 存在未处理的 P0 风险或当前阶段 P1 风险。

### 1.4 核心目标

- 解决跨项目复制时只迁页面、不迁真实调用链和异常分支的乱象。
- 统一“先资产、先底层、后业务、后页面、再适配、最后验证”的执行标准。
- 规避命名覆盖、数据/权限冲突、重复请求、埋点丢失、状态泄漏和依赖污染事故。
- 通过目标能力复用、迁移契约、分层门禁和固定 Checklist 降低重复分析与返工成本。

### 1.5 风险分级前置拦截

| 等级 | 定义 | 典型问题 | 处理规则 |
| --- | --- | --- | --- |
| P0 阻断 | 可能破坏生产、数据、安全、权限或目标已有能力 | 同名覆盖、数据结构冲突、权限绕过、路由/store/自动导入冲突、破坏性依赖变更 | 立即停止；解决并复验前禁止进入下一阶段 |
| P1 高危 | 导致主流程/关键分支异常、逻辑丢失或严重视觉不可用 | API 语义不一致、请求/埋点重复、关键状态缺失、响应式崩坏、跨页面状态污染 | 停止当前层；修复并通过当前门禁后继续 |
| P2 常规 | 不阻断核心使用的细节偏差 | 次要间距、非关键文案、低频动效、非核心极端视口瑕疵 | 记录、扣分并修复；仅经用户接受后允许进入迭代清单 |

每个风险都记录编号、证据、影响、等级、处理人/动作和状态。无法判断 P0/P1 时按更高一级处理。

## 2. 标准化流程

严格按以下九阶段顺序执行，不得跳过或并行绕过门禁。后续阶段失败时，退回最早受影响阶段，修复后重新执行其后的全部门禁。

| 阶段 | 固定动作 | 强制产出 |
| --- | --- | --- |
| 1. 梳理盘点 | 逆向源调用链；盘点目标已有能力、资产、文件、配置和风险 | 基线、调用链、UI 状态矩阵、复用候选、风险清单 |
| 2. 环境准备 | 对齐包管理器、配置、别名、运行方式、权限和验证条件 | 环境差异表、允许命令、阻塞项结论 |
| 3. 底层能力落地 | 先完成类型、API 适配、请求层、状态、基础组件、token 和资源 | 底层映射、适配层、底层验证结果 |
| 4. 业务逻辑落地 | 迁移校验、状态机、异步时序、缓存、权限、埋点和异常分支 | 业务契约实现、主/分支流程证据 |
| 5. 页面/功能复刻 | 复刻结构、文案、交互、全部 UI 状态、响应式和无障碍 | 页面实现、UI 状态对照结果 |
| 6. 差异化改造 | 复用目标能力；适配路由/布局/i18n/SEO；处理命名与目录冲突 | 适配映射、重命名映射、差异记录 |
| 7. 多层校验 | 执行静态、功能、视觉、边界和副作用检查 | 验证矩阵、失败项和修复证据 |
| 8. 回归验收 | 对照源基准和目标既有功能；计算总分并执行重做判定 | 回归报告、评分、合格结论 |
| 9. 归档沉淀 | 汇总决策、文件、风险、命令、证据、差异和后续项 | 完整交付与可追溯归档 |

逐阶段使用 [迁移 SOP 与验收模板](references/migration-contract.md) 勾选 Checklist。当前阶段 Checklist 未全部通过时，不得将下一阶段标记为进行中。

## 3. 分层执行

### 3.1 资产与环境层

从用户入口向下追踪页面/组件 → handler → composable/hook → store → service/API → 服务端/外部依赖 → 状态更新/导航/埋点。不得停在薄页面或表层组件。

同时盘点路由、布局、组件库、设计 token、图标图片、i18n、权限、实验、SEO、缓存、持久化、上传下载、轮询、测试和运行配置。记录源提交和目标起始提交，避免基线漂移。

### 3.2 底层能力层

按“复用 → 适配 → 迁入 → 阻塞”决策每项能力：

1. 先搜索目标项目现有 demo、共享 UI、业务组件、composable/hook、store、service、utils、类型和已有依赖。
2. 契约等价时直接复用；契约部分不同但可收敛时增加窄适配层。
3. 目标确实缺失时只迁入最小必要实现，不搬入整套源基础设施。
4. 缺少数据、权限、资产或授权时标记阻塞，不使用 mock、静态数据、吞错或硬编码伪装完成。

### 3.3 业务逻辑层

保持输入、输出、默认值、校验、状态转换、异步顺序、请求参数/次数、错误处理、重试/取消、缓存、刷新恢复、权限、额度、支付、实验和埋点时机一致。

检查目标生命周期是否已经执行同类请求、订阅或埋点，避免迁移后产生双重副作用。跨页面迁移时明确状态作用域、路由离开清理和页面间共享边界，禁止把页面私有状态意外提升为全局状态。

### 3.4 页面与 UI 层

覆盖初始、加载、空、成功、失败、禁用、hover、active、focus、选中、弹层、长文案、极端数据、移动端、主题和 reduced-motion 状态。

优先使用目标项目已有组件、设计 token、i18n 和资源系统。在相同数据、视口、缩放和主题下对比结构、尺寸、间距、颜色/透明度、字体层级、边框、圆角、阴影、图标、裁切、动效、键盘和触控行为。

### 3.5 差异适配与文件层

对必须迁入的新代码，尽量保留源功能的相对目录结构和文件名。仅因目标规范、已有实现或命名冲突做最小改动，并记录源路径到最终目标路径。

遇到目标同名文件时禁止覆盖：

1. 判断职责是否等价；等价时复用或最小合并目标文件，不再复制源文件。
2. 职责不等价或合并会改变既有行为时，保留目标文件并重命名迁入文件。
3. 使用业务域、功能名或职责生成语义化名称；禁止使用 `Copy`、`New`、`Temp`、`2` 等临时后缀。
4. 同步更新组件名、静态/动态导入、自动导入、路由、barrel export、类型、测试和资源引用。
5. 使用大小写敏感与不敏感搜索确认旧引用、重复导出和自动导入冲突均已消除。

## 4. 卡点校验

| 门禁 | 进入条件 | 不通过处理 |
| --- | --- | --- |
| G0 准入门禁 | 路径、权限、范围、基准、规范和写入边界明确 | 禁止开始迁移 |
| G1 资产门禁 | 调用链、UI 状态、API/状态/权限/埋点和目标复用候选已盘点 | 返回阶段 1 |
| G2 冲突门禁 | 无未解决 P0；文件、目录、路由、store、导出和自动导入映射明确 | 先重命名/适配，禁止复制 |
| G3 底层门禁 | 类型、API、状态、资源和基础组件可独立工作 | 返回阶段 2/3 |
| G4 逻辑门禁 | 主流程与关键分支行为一致，无重复副作用 | 返回阶段 3/4 |
| G5 UI 门禁 | 全部关键状态完成，同视口对照无 P1 差异 | 返回阶段 3/5/6 |
| G6 验收门禁 | 静态、功能、视觉、边界验证完成，评分达到标准 | 修复后重跑阶段 7/8 |
| G7 归档门禁 | 映射、风险、证据、命令、差异和结论可追溯 | 不得宣称交付完成 |

在每个门禁输出“通过 / 不通过 / 受限通过”。P0/P1 不允许受限通过；P2 只有记录证据、扣分并获得用户接受后才允许受限通过。

## 5. 验收标准

### 5.1 多层校验

- **静态代码校验（15 分）**：语法、类型、lint、命名/路径冲突、残留引用、重复导出、项目规范和无关改动。
- **功能逻辑校验（35 分）**：主流程、关键分支、API 契约、状态转换、权限、缓存、埋点和请求次数。
- **视觉还原校验（30 分）**：结构、几何、视觉、交互、状态、响应式和动效；有 URL 时使用截图与 computed style 对照。
- **边界场景校验（15 分）**：空数据、异常、慢请求、超时、取消、重复提交、快速切换、刷新恢复、极端内容和并发。
- **归档完整度（5 分）**：映射、风险、证据、命令、差异、评分和后续项完整。

### 5.2 量化判定

| 结论 | 分数与硬条件 | 动作 |
| --- | --- | --- |
| 合格 | 95–100 分；P0=0、P1=0；主流程、关键分支、关键 UI 状态和冲突检查全部通过 | 可交付；P2 仅在用户接受后进入迭代清单 |
| 不合格 | 80–94 分，或存在未完成的必要验证但无 P0/P1 | 修复扣分项并重新执行阶段 7/8，不得标记完成 |
| 重做 | <80 分，或存在任一 P0、未解决 P1、主流程失败、业务逻辑丢失、权限/数据契约错误、同名覆盖、未授权依赖变更 | 退回最早错误阶段，重做其后全部阶段 |

无法运行源或目标功能、缺少关键视觉基准、未做同条件对照时，不得获得“UI 一比一通过”结论；只能标记“代码级迁移完成，运行时待验收”。详细评分逐项记录在迁移模板中，不凭主观印象打分。

## 6. 故障兜底

按“保护目标既有能力 → 冻结当前层改动 → 定位最早错误阶段 → 使用最小修复 → 重跑后续门禁”处理故障。禁止通过覆盖文件、扩大重构、吞错、禁用校验、删除测试、加入临时后缀或修改无关代码绕过问题。

高频故障按以下顺序排查：

1. 文件/目录/路由/store/自动导入同名冲突。
2. 目标已有实现与源契约不完全等价。
3. API、鉴权、错误格式或数据结构不一致。
4. 生命周期导致重复请求、重复订阅或重复埋点。
5. 跨页面状态未清理、串页或刷新丢失。
6. i18n、资源路径、设计 token、布局或响应式适配缺失。
7. 类型/lint/构建失败混入目标项目既有错误。
8. 源/目标无法运行，缺少视觉或交互证据。

命中故障后读取 [高频故障兜底手册](references/failure-playbook.md)，按症状、证据、根因、最小修复和复验项执行。连续两次修复仍失败时升级风险等级；达到 P0/P1 后停止并报告阻塞，不继续试错扩大影响面。

## 7. 归档规范

最终交付必须包含：

- 任务范围、源/目标路径、基准分支/提交和验收环境。
- 源调用链、目标复用能力、API/状态/权限/埋点适配关系。
- 源路径到目标路径、同名冲突、重命名和全部引用更新记录。
- P0/P1/P2 风险及其证据、处理、复验和最终状态。
- 静态、功能、视觉、边界验证命令/环境/结果与 100 分制评分。
- 用户可感知影响、下游消费者影响、性能影响和回滚关注点。
- 已接受 P2、未完成项、跳过检查、无法验证项及原因。
- 完整检索范围和每阶段 Checklist 结论。

除非用户或项目规范要求，不把临时迁移契约写入产品仓库。需要永久归档时，遵循目标项目文档目录和命名规范，默认使用 `YYYY-MM-DD-<source>-to-<target>-<feature>.md`，不得包含密钥、cookie、token、个人数据或生产敏感响应。

不得主动暂存或提交。用户明确要求时，只暂存本次迁移文件，按目标仓库提交规范创建原子提交，并在提交前重新执行 G6/G7 门禁。
