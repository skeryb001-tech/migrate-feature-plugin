## 统一止损步骤

1. 停止当前阶段和其后的写入，不继续堆补丁。
2. 保护目标已有实现和用户改动，不覆盖、不回退、不删除未知文件。
3. 记录最小复现、错误文本、调用链、Network/日志和相关 diff。
4. 确定最早出错阶段：资产、环境、底层、业务、页面或适配。
5. 按最小边界修复，不借机做无关重构。
6. 重跑该阶段及其后的全部门禁、评分和回归。
7. 连续两次修复失败时提高一级风险；升级为 P0/P1 后停止并报告。

## 1. 同名文件或自动导入冲突

- **症状**：构建解析到错误组件；路由/store 重复；自动导入重名；目标原功能被替换。
- **风险**：P0。
- **定位**：检查精确同名、大小写同名、目录名、组件名、路由名、store id、导出名和框架生成的自动导入名。
- **最小修复**：职责等价时复用/合并目标文件；职责不同则保留目标文件，用业务域/功能/职责重命名迁入文件。
- **禁止**：覆盖目标文件；使用 `Copy`、`New`、`Temp`、`2`；只改文件名不改自动导入或动态导入。
- **复验**：执行大小写敏感/不敏感搜索、类型/构建检查，并手动验证两个功能入口。

## 2. 目标已有实现与源契约不等价

- **症状**：目标组件能显示但缺少事件、状态、插槽、错误分支或副作用。
- **风险**：关键流程受影响为 P1，否则 P2。
- **定位**：逐项对比 props/input、emit/output、默认值、状态、时序、错误和样式状态，不按文件名判断等价。
- **最小修复**：优先增加薄适配层或扩展目标公共能力；只有目标能力无法安全扩展时迁入缺失部分。
- **复验**：重跑调用链契约、UI 状态矩阵和目标已有消费者回归。

## 3. API、鉴权或数据结构不一致

- **症状**：页面可见但请求失败、字段缺失、错误被吞、权限错误或数据写入异常。
- **风险**：数据/权限冲突为 P0，其余关键流程为 P1。
- **定位**：对比方法、URL、headers、鉴权、参数、序列化、响应、错误码、超时、重试、取消和环境代理。
- **最小修复**：在 service/API 边界增加类型明确的适配器；保持 UI 使用统一业务契约。
- **禁止**：在组件内散落字段兼容；使用静态数据假装成功；修改生产协议但不评估下游。
- **复验**：用相同输入对照请求/响应和失败分支，确认无额外写入。

## 4. 重复请求、订阅、计时器或埋点

- **症状**：Network 出现双请求；积分/数据重复扣减；事件发送两次；页面离开后仍轮询。
- **风险**：涉及计费/写入为 P0，其余为 P1。
- **定位**：检查页面、子组件、hook/store 初始化和生命周期是否同时执行相同副作用。
- **最小修复**：保留唯一职责所有者；删除重复预取/监听；补齐取消、销毁和幂等保护。
- **复验**：统计一次完整操作的请求、订阅、计时器和埋点次数，并验证快速切页。

## 5. 跨页面状态串页或刷新丢失

- **症状**：B 页面继承 A 页面残留；切页后状态未清理；刷新后关键结果消失。
- **风险**：错误用户数据/权限泄漏为 P0，功能异常为 P1。
- **定位**：确认状态属于节点/组件、页面、会话还是全局；检查初始化、路由 key、持久化和 unmount 清理。
- **最小修复**：把状态放回正确作用域；按业务标识隔离；只持久化可序列化且耐久的源数据；补齐恢复/loading。
- **复验**：覆盖 A→B→A、刷新、后退、并行标签页和不同业务 id。

## 6. 路由、布局、SSR 或 hydration 异常

- **症状**：页面布局错位、仅客户端正常、刷新报错、组件重复挂载或 hydration warning。
- **风险**：页面不可用为 P1。
- **定位**：对比路由参数、layout、middleware、SSR 边界、client-only API、自动导入和服务端初始数据。
- **最小修复**：按目标项目路由/layout 约定接入；把浏览器 API 放入正确客户端生命周期；保持服务端和客户端初始状态一致。
- **复验**：直接 URL、站内导航、刷新、后退、SSR 输出和控制台均验证。

## 7. UI 视觉偏差反复修不准

- **症状**：颜色、尺寸、间距或对齐多次调整仍不一致；只在某视口正确。
- **风险**：关键交互不可用为 P1，纯细节为 P2。
- **定位**：固定相同数据、视口、缩放、主题和字体加载状态；查看 computed style、盒模型、资源固有尺寸和父级约束。
- **最小修复**：先使用目标设计 token 和已有组件状态；修父级布局后再修子级；逐项对照结构、几何、视觉、交互和动效。
- **禁止**：无依据堆任意值、`!important`、全局 CSS 或仅凭肉眼反复猜测。
- **复验**：使用真实浏览器记录引擎版本、关键视口截图、CSSOM computed style 和几何测量，并确认关键盒模型误差 ≤1 CSS px；自定义渲染器仅作补充。

## 8. i18n、长文案或多语言布局破坏

- **症状**：硬编码文案、缺少语言、按钮溢出、换行错位或服务端/客户端语言不一致。
- **风险**：核心操作不可识别为 P1，其余为 P2。
- **定位**：检查目标语言清单、命名空间、动态加载、复数/变量和最长语言文案。
- **最小修复**：接入目标 i18n；补齐全部目标语言；使用已有截断、换行和弹性布局模式。
- **复验**：逐语言验证关键视口、长文本、空值和变量替换。

## 9. 路径大小写、barrel export 或旧引用残留

- **症状**：本机正常、CI/Linux 失败；导出解析到旧文件；删除后仍被间接引用。
- **风险**：构建/运行失败为 P1。
- **定位**：分别执行精确、忽略大小写和旧名称搜索；检查 index/barrel、动态字符串导入、测试和生成配置。
- **最小修复**：统一真实文件名与引用大小写；更新/删除旧导出；避免循环依赖。
- **复验**：类型、构建、测试和仓库级旧名称搜索全部通过。

## 10. 依赖或组件 API 版本不匹配

- **症状**：复制示例无法编译；组件 props/events 不存在；运行行为与源项目不同。
- **风险**：P1；涉及破坏性依赖变更时 P0。
- **定位**：读取两个项目 `package.json`、锁文件和组件库实际版本，查对应版本文档与本地用法。
- **最小修复**：优先使用目标版本已有 API 或项目封装；需要依赖变更时先提交风险和迁移方案并等待授权。
- **复验**：聚焦组件测试、类型检查、目标现有消费者回归。

## 11. 源或目标无法运行

- **症状**：开发服务不可启动、环境变量缺失、账号/接口不可用或无访问权限。
- **风险**：无法完成的关键验证为 P1，不等于代码失败。
- **定位**：区分环境阻塞、项目既有错误和迁移错误；保留原始错误证据。
- **最小修复**：使用现有测试、静态调用链、设计稿、截图或 mock server 等已授权证据；不要伪造运行结果。
- **复验**：能执行多少验证就执行多少，并明确剩余风险。
- **交付限制**：报告使用 `runtime_visual_verified: NO`、G6/G8=`PENDING_RUNTIME`、`final_conclusion: CODE_ONLY`；校验脚本应返回 3，不得声明一比一验收通过。

## 12. 脏工作区或既有检查错误干扰

- **症状**：目标文件已有修改；全量 lint/typecheck 报大量无关错误；diff 难以归因。
- **风险**：覆盖用户改动为 P0，验证噪声为 P2/P1。
- **定位**：迁移前记录 `git status`；按文件和错误路径区分本次与既有问题。
- **最小修复**：只合并本次范围；优先聚焦检查；标注既有错误但不顺手修复。
- **禁止**：reset/checkout 用户改动；清空暂存区；为了通过检查修改无关模块。
- **复验**：确认本次新增错误为 0，最终文件清单只包含任务范围。

## 13. 性能退化或竞态

- **症状**：输入卡顿、重复渲染、深监听高耗、请求乱序覆盖、切页后更新已销毁组件。
- **风险**：关键流程错误为 P1，否则 P2。
- **定位**：检查 watcher、列表、计算、请求并发、取消信号、轮询、资源懒加载和组件生命周期。
- **最小修复**：复用目标项目节流/防抖/缓存/取消模式；缩小监听范围；保证最新请求或业务 id 才能写状态。
- **复验**：快速操作、慢网、并发、离开页面和大数据量场景。

## 14. 权限、支付、实验或埋点语义丢失

- **症状**：未登录可执行受限操作；付费门槛消失；实验分流变化；埋点时机或字段改变。
- **风险**：权限/支付为 P0，关键实验/埋点为 P1。
- **定位**：追踪实际 handler 到权限/支付/实验/埋点入口，不根据函数名猜测；核对触发边缘和下游消费者。
- **最小修复**：复用目标中心化能力；保持业务触发时机和 payload 语义；契约变化需明确下游迁移。
- **复验**：覆盖登录状态、权限档位、支付取消/失败、实验分组和事件次数/载荷。

## 15. 资源、上传下载或持久化失效

- **症状**：图标/图片 404；上传后刷新丢失；blob/File 被错误持久化；下载名或类型错误。
- **风险**：数据丢失或错误写入为 P0，功能不可用为 P1。
- **定位**：检查资源别名、公开路径、构建处理、上传接口、远程 URL、session id、序列化和恢复时序。
- **最小修复**：接入目标资源与上传封装；只持久化耐久标识和远程 URL；恢复时提供明确 loading/error 状态。
- **复验**：上传、刷新、重新进入、失败重试、跨会话和下载结果均验证。

## 16. 错误保留源结构导致目标架构失配

- **症状**：迁移代码可以运行，但在目标项目中出现第二套 components/hooks/stores/services 目录、重复基础设施、跨层导入、命名不一致或测试无法按目标方式组织。
- **风险**：破坏公共边界、自动导入或既有消费者时为 P0；造成长期重复能力或关键维护风险时为 P1；纯非阻断组织瑕疵为 P2。
- **定位**：对照目标项目规范、相邻模块、demo、共享能力和目录所有权；逐项检查源职责在目标架构中的正确落点，而不是比较源/目标路径是否相同。
- **根因**：错误地把“一比一迁移”理解为文件和目录一比一复制，或把“尽量保留源结构”放在目标规范之前。
- **最小修复**：保持对外业务契约和 UI 结果不变，先复用目标能力，再把迁入代码重组到目标规定的页面、业务、共享 UI、状态、请求、类型和测试层；仅在符合目标规则且无冲突时保留源文件名。
- **同名处理**：目标同名职责等价则复用/最小合并；职责不同则保留目标文件并按目标命名规范语义化重命名迁入文件，更新全部静态、动态和自动导入。
- **消费者保护**：移动或调整公共模块前扫描全部直接、间接、动态和自动导入消费者，默认保持兼容并完成聚焦回归。
- **复验**：目标规范检查、目录/分层映射、循环依赖、自动导入、lint/typecheck、构建、源行为对照和关键消费者回归全部通过。
