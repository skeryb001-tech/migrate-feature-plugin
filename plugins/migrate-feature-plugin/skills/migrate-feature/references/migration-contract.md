# 前端功能迁移 SOP 与验收模板

在任务计划或临时工作记录中填写本模板。除非用户或目标项目规范要求，不把模板原样复制为产品仓库永久文档。

固定决策顺序为：源行为一致 → 目标能力复用 → 目标代码规范与结构 → 仅在兼容时参考源结构与文件名。UI 结果以源功能为准，迁移后的目录、分层、命名和实现方式以目标项目为准。

可先运行 `scripts/createMigrationReport.py` 生成模式专属报告，最终用 `scripts/validateMigrationReport.py` 执行最低验收门禁。

## 10. 计分与证据规则

### 10.1 默认 UI 误差

在相同数据、视口、设备像素比、缩放、主题、语言和字体加载状态下：

- 关键组件的宽高、间距、对齐和位置误差必须 ≤1 CSS px。
- 颜色和透明度必须一致；复用目标设计 token 时，computed style 和最终视觉必须等价。
- 排除字体栅格化、抗锯齿和操作系统字体渲染差异后，非文本像素差异默认必须 ≤0.5%。
- 字体抗锯齿可以排除，但文字容器、行高、换行、基线和整体盒模型不能排除。
- 动效需比较触发条件、时长、缓动、关键状态、打断和 reduced-motion，不能只比较静态首尾帧。

若项目或用户给出更严格阈值，使用更严格值；放宽阈值必须记录为 P2/P1 并由用户明确接受。

### 10.2 证据有效性

有效证据必须能让另一位执行者复现结论，例如命令及输出、测试用例、Network 记录、调用链、截图、computed style、像素 diff、消费者搜索或代码 diff。

- 仅写“已检查”“看起来一致”“应该没问题”不算证据。
- 无法运行检查时保留原始阻塞证据，并把对应评分项记 0。
- `N/A` 不是跳过校验的手段；必须解释为什么业务上不适用，并给出证明材料。
- 源或目标无法运行、没有视觉基准或未完成同条件对照时，不得宣称 UI 一比一通过。
